# gstack 技能完全操作手册

> **版本**: 0.3.3 | **更新日期**: 2026-03-20
> 
> 本手册涵盖 gstack 所有技能的完整说明、使用场景与操作示例。

---

## 技能总览

gstack 的技能按功能分为五大类：

| 类别 | 技能 | 核心角色 |
|------|------|----------|
| **规划** | `/office-hours` `/plan-ceo-review` `/plan-eng-review` `/plan-design-review` | 在写代码前把方向定对 |
| **设计** | `/design-consultation` `/design-review` | 从零构建设计系统，审查视觉质量 |
| **开发质量** | `/review` `/investigate` `/codex` | 代码审查、调试、多模型交叉验证 |
| **测试与发布** | `/qa` `/qa-only` `/browse` `/setup-browser-cookies` `/ship` `/document-release` `/retro` | 测试、发布、写文档、复盘 |
| **安全防护** | `/careful` `/freeze` `/guard` `/unfreeze` `/gstack-upgrade` | 防误操作、限制编辑范围、升级工具 |

```
办公时间 → 战略规划 → 技术规划 → 设计规划 → 开发实现 → 代码审查 → QA测试 → 发布 → 文档更新 → 周复盘
/office-hours → /plan-ceo-review → /plan-eng-review → /plan-design-review → [coding] → /review → /qa → /ship → /document-release → /retro
```

## 一、规划类技能

### `/office-hours` — YC 创始人办公室

**角色**: YC Office Hours 合伙人  
**使用时机**: 有新想法，在写任何代码之前

#### 核心价值

这是整个 gstack 工作流的起点。它的工作不是帮你实现你描述的功能，而是质疑你的描述，找到你真正应该构建的东西。

**真实案例**: 用户说要做"日程管理 App"。经过 office-hours 的追问，发现用户真正需要的是"AI 私人助理——自动处理跨日历账号冲突、生成真实的会议准备材料、主动管理联系人 CRM"。这个重定向改变了整个产品方向。

#### 两种模式

**创业模式**（Startup Mode）：适合正在做公司或内部项目的人。会用 YC 的六个追问框架逼你找到真正的需求：
1. **需求现实** — 你有什么证据证明有人真的需要这个？（不是"觉得不错"，是真的会因为没有它而痛苦）
2. **现状调研** — 用户现在怎么凑合解决这个问题的？代价是什么？
3. **最具体的用户** — 说出那个最需要你产品的人是谁，职位，公司，什么让他夜里睡不着？
4. **最小楔子** — 最小的、用户愿意这周就付钱的版本是什么？
5. **观察与惊喜** — 你真的坐在用户旁边看他用过你的产品吗？什么让你意外？
6. **未来适配性** — 3年后世界变了，你的产品是更重要还是更不重要了？

**Builder 模式**：适合黑客马拉松、Side Project、开源项目、学习项目。是热情的合作者，帮你找到想法里最酷的版本。不是质问而是共创。

#### 输出结果

会话结束后产出设计文档，写入 `~/.gstack/projects/`，可被后续 `/plan-ceo-review`、`/plan-eng-review` 自动读取。

#### 使用示例

```
"我想做一个 AI 辅助的求职简历工具"  →  /office-hours

→ 会问：谁最需要这个？现在怎么做简历的？愿意付多少钱？
→ 可能重框架为：不是简历工具，是面试准备系统
→ 产出设计文档
```

### `/plan-ceo-review` — CEO 战略视角审查

**角色**: 创始人 / CEO，像 Brian Chesky 一样思考  
**使用时机**: 有了明确的功能计划，想在实现前确保方向正确、目标足够宏大

#### 核心价值

不是帮你把功能做出来，而是追问：**这个计划包含了哪个十星级产品？**

**示例**: 
- 你说："给卖家加一个上传商品图的功能"
- `/plan-ceo-review` 会问：上传图片只是手段，真正的目标是让商品卖出去。那么——能不能自动识别商品？自动抓取产品规格？自动生成高转化率的标题和描述？自动判断哪张图片最能促成购买？自动检测图片是否暗淡、杂乱、低信任感？
- 这一追问，把一个"上传框"变成了一个"智能上架助手"。

#### 四个模式

| 模式 | 适用场景 | 行为 |
|------|----------|------|
| **SCOPE EXPANSION** 扩大范围 | 全新功能，想做大 | 热情推进野心版，每个扩展逐一确认 |
| **SELECTIVE EXPANSION** 选择性扩展 | 想守住主线，但愿意看看还能加什么 | 保持现有范围，中性提供扩展候选，逐一决策 |
| **HOLD SCOPE** 锁定范围 | 方向已定，只要求严格审查 | 不扩展，最大力度审查架构、安全、测试 |
| **SCOPE REDUCTION** 缩减范围 | 计划超载，要找最小可行版本 | 外科手术式切除，找到能交付核心价值的最小版本 |

#### 审查内容（共10大板块）

- 架构审查（依赖图、数据流四路径）
- 错误与异常映射
- 安全与威胁模型
- 数据流与边界情况
- 代码质量审查
- 测试计划审查
- 性能审查
- 可观测性与调试性
- 部署与回滚方案
- 长期技术债务影响

### `/plan-eng-review` — 工程经理技术审查

**角色**: 最好的技术 Lead  
**使用时机**: 产品方向对了，需要把模糊想法变成可执行的技术方案

#### 核心价值

和 `/plan-ceo-review` 的"梦想"不同，这里的任务是**让想法真正可以构建**。回答的问题是：架构边界在哪？哪些步骤同步哪些异步？失败的处理链路是什么？

最独特的价值：**强制画图**。架构图、数据流图、状态机图、测试矩阵。LLM 在需要画图的时候，会把隐藏的假设都逼出来，让模糊的规划变得无处遁形。

#### 三大门禁系统

每次规划审查（CEO/Eng/Design）都会记录结果，形成 **Review Readiness Dashboard**：

```
+====================================================================+
|                    REVIEW READINESS DASHBOARD                       |
+====================================================================+
| Review          | Runs | Last Run            | Status    | Required |
|-----------------|------|---------------------|-----------|----------|
| Eng Review      |  1   | 2026-03-16 15:00    | CLEAR     | YES      |
| CEO Review      |  1   | 2026-03-16 14:30    | CLEAR     | no       |
| Design Review   |  0   | —                   | —         | no       |
+--------------------------------------------------------------------+
| VERDICT: CLEARED — Eng Review passed                                |
+====================================================================+
```

**Eng Review 是唯一的必须门禁**（可配置跳过）。CEO 和 Design 是推荐项。

#### Plan-to-QA 流

`/plan-eng-review` 完成测试审查后，自动写出测试计划到 `~/.gstack/projects/`。当你后续运行 `/qa` 时，它会自动读取这个测试计划——工程审查直接喂给 QA 测试，无需手动传递。

### `/plan-design-review` — 设计规划审查

**角色**: 资深设计师（规划阶段）  
**使用时机**: 有带 UI 的功能计划，在实现之前做设计检查

#### 核心价值

大多数计划只描述后端做什么，完全不说用户实际看到什么。空状态是什么？加载状态是什么？移动端怎么显示？这些都被推迟到"实现时再说"——然后工程师随手写了个 "No items found."

`/plan-design-review` 在规划阶段就找出这些漏洞，修改计划文档，当时就解决。

#### 七个检查维度（0-10 评分）

1. **信息架构** — 用户先看什么？再看什么？第三看什么？
2. **交互状态覆盖** — 每个功能 × 5 种状态（加载/空/错误/成功/部分）是否都有说明
3. **用户旅程连贯性** — 情感弧线是否完整？
4. **AI 千篇一律风险** — 有没有"渐变 hero"、"三列图标网格"、"圆角卡片堆叠"这类 AI 套模板的迹象
5. **设计系统对齐** — 是否符合项目的 DESIGN.md
6. **响应式和无障碍** — 移动端、键盘导航、对比度
7. **未解决设计决策** — 凡是"实现时再说"的设计点，全部标出

#### 示例输出

```
/plan-design-review

初始设计评分：4/10

"此计划描述了用户仪表板，但未指定用户首先看到什么。
 它说'带图标的卡片'，这和所有 SaaS 模板一样。
 没有任何加载状态、空状态描述，也没有移动端行为说明。"

修复后评分：4/10 → 8/10
"计划已具备设计完整性。实现后请运行 /design-review 进行视觉 QA。"
```

## 二、设计类技能

### `/design-consultation` — 从零构建设计系统

**角色**: 设计合伙人  
**使用时机**: 项目没有设计系统、字体、配色时，从零开始建立视觉身份

#### 核心价值

不是填表格，而是真正的对话式设计咨询。AI 了解你的产品、用户、受众，推导出这个产品需要传达什么感觉（可信赖感、速度感、工艺感、温暖感），然后倒推到具体选择：字体（含具体用途角色分配）、色盘（含十六进制值）、间距尺度、布局策略、动效策略。

**关键点：大胆的创意冒险**。不只是推荐"安全选择"（同类产品都这样），还会明确推荐"创意风险"（偏离惯例但更有记忆点的选择），并告诉你哪个是哪个，让你决定。

#### 竞品景观研究

可以先截图分析你所在品类的竞品网站（字体、配色、间距），让你看清惯例是什么，然后才决定哪些值得打破。

#### 输出结果

- 交互式 HTML 预览页面——真实产品页面（仪表板、营销页）完整渲染你的设计系统
- `DESIGN.md` 写入项目根目录，成为设计的唯一真实来源
- `CLAUDE.md` 自动更新，此后所有会话都遵循这套设计系统

#### 示例

```
/design-consultation

Claude: 从 README 看，这是一个面向工程团队的开发者分析仪表板。我来研究一下
        这个品类现有的产品长什么样——[浏览 Linear, Vercel, Datadog, Posthog]

        它们都收敛在 sans-serif 和灰蓝色调。Linear 靠深色优先和紧凑间距突出。
        其余的几乎无法区分。

        我的提案：
        工业/实用主义 — 数据优先，每个像素都必须有价值
        字体：Geist（正文+UI），Instrument Serif（展示），JetBrains Mono（数据）
        配色：克制 — 石板中性色，单一青色强调（#0D9488）

        安全选择 vs 创意风险：
        ✅ Geist 正文 — 用户已从 Vercel 熟悉此字体
        🎲 展示标题用 Serif — 开发工具里没人用；瞬间建立层次感
        🎲 青色非蓝色 — 这个品类的人都用蓝色；青色够近不失可信，够远足以被记住

您想保留哪些创意风险？
```

### `/design-review` — 实时网站视觉审计

**角色**: 会写代码的设计师  
**使用时机**: 实现完成后，对运行中的网站进行视觉 QA 并自动修复

#### 核心价值

`/plan-design-review` 审查计划，`/design-review` 审查已上线的网站。  
运行 80 项视觉审计，然后进入修复循环：找到源文件 → 最小改动 → 提交（`style(design): FINDING-NNN`）→ 截图验证，每个问题一个提交，完全可 bisect。

#### 示例

```
/design-review https://myapp.com

Claude: 运行 80 项视觉审计中……
Design Score: C  |  AI Slop Score: D
12 个发现（4 高优先级，5 中，3 润色）

自动修复进行中……

style(design): FINDING-001 — 三列图标网格改为非对称布局
style(design): FINDING-002 — 建立标题尺度 48/32/24/18/16
style(design): FINDING-003 — 移除渐变 hero，改用粗体排版
style(design): FINDING-004 — 为标题引入第二字体
style(design): FINDING-005 — 按元素角色区分圆角值
style(design): FINDING-006 — 正文左对齐，居中仅用于标题
style(design): FINDING-007 — 为所有交互元素添加 hover/focus 状态
style(design): FINDING-008 — 添加 prefers-reduced-motion 媒体查询
style(design): FINDING-009 — 设置正文最大宽度 680px

最终审计：
Design Score: C → B+  |  AI Slop Score: D → A
9 个修复已应用（8 个验证通过，1 个尽力而为）。3 个延后处理。
```

## 三、开发质量类技能

### `/review` — 生产级代码审查

**角色**: 偏执的 Staff 工程师  
**使用时机**: 合并 PR 之前，找出那些能通过 CI 但在生产爆炸的 Bug

#### 核心价值

通过 CI 不代表分支安全。`/review` 关注的是 CI 根本测不出来的问题：

- **N+1 查询**（循环里隐藏的数据库调用）
- **竞态条件**（两个标签页同时操作同一资源）
- **信任边界违规**（AI 输出在未验证的情况下写入数据库）
- **枚举完整性**（添加了新的状态常量，但没有更新所有处理该状态的 switch 语句）
- **遗漏索引**（新的查询字段没有对应的数据库索引）
- **孤儿资源**（上传成功但处理失败，文件永久留在存储中）

#### Fix-First 模式

不只是列出问题。自动修复显而易见的机械性问题（死代码、过时注释、N+1），真正需要决策的问题才问你。

```
Pre-Landing Review: 7 个发现（2 关键，5 参考）

[AUTO-FIXED] app/models/item.rb:142 — 移除 3 层循环中的 N+1 查询 → 改用 includes
[AUTO-FIXED] app/services/uploader.rb:88 — 删除遗留的调试日志

需要您确认：
1. [CRITICAL] app/services/payment.rb:47 — 并发时可能重复扣费，无锁保护
   → A) 修复  B) 跳过

2. [CRITICAL] app/models/listing.rb:22 — 未对 AI 返回的 JSON 进行 schema 验证就写入 DB
   → A) 修复  B) 跳过
```

#### 作用域漂移检测

审查还会比对"PR 描述说要做什么"与"diff 实际改了什么"，标出超出范围的改动（范围蔓延）和未完成的需求（缺失实现）。

### `/investigate` — 系统性根因调试

**角色**: 调试专家  
**使用时机**: 发生了 Bug，但不知道为什么

#### 铁律：没有根因，不准修复

`/investigate` 遵循严格的调查优先原则：不猜测、不盲目修补。流程是：

1. **调查阶段** — 追踪数据流，复现问题，收集证据
2. **分析阶段** — 与已知 Bug 模式匹配，找出最可能的根因
3. **假设阶段** — 提出可测试的假设（每次只测一个）
4. **实现阶段** — 确认根因后才修复，若连续 3 次修复失败则回到分析阶段

`/freeze` 会自动激活，限制编辑范围在正在调试的模块内，防止误改无关代码。

---

### `/codex` — 多 AI 交叉代码审查

**角色**: 来自 OpenAI Codex 的第二意见  
**使用时机**: 想要用不同 AI 模型交叉验证同一份代码

#### 三种模式

**Review 模式**: 用 `codex review` 审查当前 diff，得出 PASS/FAIL 结果。任何 P1 问题 = FAIL。审查完全独立，Codex 看不到 Claude 的审查结果。

**Challenge 对抗模式**: Codex 主动尝试破坏你的代码。寻找边界情况、竞态条件、安全漏洞、在高负载下会失败的假设。使用最高推理力度（xhigh）。相当于对代码逻辑做渗透测试。

**Consult 咨询模式**: 带会话连续性的开放对话。向 Codex 询问任何关于代码库的问题，后续问题复用同一会话，上下文持续传递。

#### 跨模型分析

当 `/review`（Claude）和 `/codex`（OpenAI）都检查过同一分支时，会产生交叉比对报告：

```
/codex review

CODEX REVIEW: PASS（3 个发现）
[P2] 支付处理器中的竞态条件 — 并发请求可能重复扣费
[P3] user.email 在 downcase 前缺少 null 检查
[P3] Token 比对没有使用常量时间比较

跨模型分析（vs /review）：
OVERLAP: 支付竞态条件（两者都发现）← 高置信度
UNIQUE TO CODEX: Token 比较定时攻击
UNIQUE TO CLAUDE: listing 图片 N+1 查询
```

两个 AI 都发现的问题：高置信度真实 Bug。只有其中一个发现的：盲点揭示。

## 四、测试与发布类技能

### `/browse` — 无头浏览器

**角色**: 给 AI 搭配一双眼睛  
**使用时机**: 任何需要查看 UI 状态、测试交互、截图的场景

#### 核心能力

- 基于 Playwright + 真实 Chromium，非模拟
- 首次启动约 3 秒，后续命令约 100-200ms
- 跨命令保持状态（Cookie、标签页、localStorage）
- 支持表单填写、点击、截图、控制台日志读取

#### 常用命令格式

```bash
browse goto https://myapp.com          # 导航到URL
browse snapshot -i                     # 截图并标注交互元素
browse fill @e2 "user@example.com"    # 向元素填值（@e2 是快捷引用）
browse click @e5                       # 点击元素
browse screenshot /tmp/result.png      # 保存截图
browse console                         # 读取控制台日志
browse handoff "卡在了验证码"           # 移交给真人浏览器
browse resume                          # AI 重新接管
```

#### 浏览器移交（Handoff）

当遇到验证码、多因素认证等 AI 无法处理的情况，可以无缝移交给用户：

```
Claude: 我在登录页卡在了验证码。正在打开可视 Chrome 浏览器。

        > browse handoff "登录页验证码"

        Chrome 已打开 https://app.example.com/login，所有 Cookie 和标签页都保留了。
        请解决验证码，完成后告诉我。

用户:   好了

Claude: > browse resume

        已获取新快照。已成功登录。继续 QA。
```

### `/setup-browser-cookies` — 导入真实浏览器 Cookie

**角色**: 会话管理器  
**使用时机**: 在运行 `/qa` 或 `/browse` 之前，导入真实登录状态

这个技能从你真实的浏览器（Chrome、Arc、Brave、Edge）导入 Cookie，让 AI 测试登录后的页面，而不需要每次重新登录。

```
/setup-browser-cookies
→ 弹出域名选择界面，选择你需要的域名
→ 导入完成：47 个 Cookie 已导入

/setup-browser-cookies github.com
→ 直接导入，无需 UI：已从 Chrome 导入 12 个 Cookie
```

**安全说明**: Cookie 明文仅存在内存中，不落盘。通过 macOS Keychain 解密，需用户明确授权。

### `/qa` — 自动 QA 测试并修复

**角色**: QA 负责人  
**使用时机**: 功能完成后，在合并前全面 QA

#### 四种模式

| 模式 | 适用场景 | 耗时 |
|------|----------|------|
| **Diff-aware 差异感知**（Feature 分支默认）| 读取 git diff，只测被改动的页面和流程 | 1-5 分钟 |
| **Full 全面扫描** | 系统性探索整个 App，产出 5-10 个有据可查的问题 | 5-15 分钟 |
| **Quick 快速冒烟**（`--quick`）| 首页 + 顶部5个导航目标 | 30 秒 |
| **Regression 回归对比**（`--regression baseline.json`）| 全面扫描，然后与之前的基线 JSON 对比 | 5-15 分钟 |

#### 自动生成回归测试

`/qa` 修复一个 Bug 并验证后，自动生成一个捕获该场景的回归测试，防止同样问题再次出现。

#### 三个严重级别

| 级别 | 说明 |
|------|------|
| **Critical/High** | 功能完全损坏或数据丢失风险，必须修复 |
| **Medium** | 体验受损，应该修复 |
| **Cosmetic** | 视觉问题，可选修复 |

```
/qa https://staging.myapp.com

Claude: [探索 12 个页面，填写 3 个表单，测试 2 个流程]

QA 报告：staging.myapp.com — 健康评分：72/100

前 3 个问题：
1. 严重：结账表单可在空必填字段下提交
2. 高：移动端导航菜单选择后不关闭
3. 中：仪表板图表在 1024px 以下与侧边栏重叠

自动修复2个问题……
[报告与截图已保存至 .gstack/qa-reports/]
```

### `/qa-only` — 仅报告，不修复

与 `/qa` 完全相同的测试方法，但只产出报告，不做任何代码修改。适合需要纯 Bug 报告、不希望 AI 自动改代码的场景。

### `/ship` — 一键完整发布流程

**角色**: 发布工程师  
**使用时机**: 代码准备好，要合并推送创建 PR

#### 完整流程

1. 检测并合并 base 分支（防止落后）
2. 检查 Review Readiness Dashboard，确认工程审查是否通过
3. 运行测试（没有测试框架的会自动搭建）
4. 构建代码覆盖率审计，自动补充测试
5. Pre-landing code review
6. 读取 Greptile 的自动审查评论并分类处理
7. 更新 VERSION 和 CHANGELOG
8. Push 并创建/更新 PR

#### 测试框架引导

如果项目没有测试框架，`/ship` 会：检测运行时环境 → 选择最佳框架 → 安装 → 为现有代码写 3-5 个真实测试 → 配置 GitHub Actions CI → 创建 TESTING.md

#### 覆盖率报告

每次 `/ship` 都会产出覆盖率图：

```
Tests: 42 → 47 (+5 new)
Coverage diagram:
app/models/listing.rb     ████████████░░░░ 75% ⭐⭐⭐
app/services/uploader.rb  ████░░░░░░░░░░░░ 25% ⭐
```

### `/document-release` — 发布后文档同步

**角色**: 技术写作者  
**使用时机**: `/ship` 创建 PR 后，在合并前同步所有文档

读取所有项目文档（README、ARCHITECTURE、CONTRIBUTING、CLAUDE.md 等），与 diff 交叉比对，自动更新过时的文件路径、命令列表、项目结构说明等。风险较高或主观性强的改动会单独询问。

```
/document-release

Claude: 分析了 3 次提交中的 21 个改动文件。找到 8 个文档文件。

        README.md：技能数量从 9 更新为 10，在表格中添加新技能
        CLAUDE.md：在项目结构中添加了新目录
        CONTRIBUTING.md：当前——无需改动
        TODOS.md：标记 2 个已完成项，添加 1 个新项

        所有文档已更新并提交。PR 描述已更新文档差异。
```

### `/retro` — 工程周复盘

**角色**: 工程经理  
**使用时机**: 每周末，看清这周真正发生了什么

#### 数据驱动，非感觉驱动

分析提交历史、工作模式、发货速度，产出坦诚的复盘报告：

- 每人的提交数、LOC、测试比、PR 大小、修复率
- 从提交时间戳检测编码会话，找出热点文件
- 跟踪发货连续天数，找到本周最大发货
- 测试健康报告（测试文件总数、本期新增测试、回归测试）

#### 团队感知

识别执行命令的人，对你自己做最深入的分析，然后为每个贡献者给出具体的表扬和成长建议。

```
/retro

Claude: 3月第1周：47次提交（3人），3.2k LOC，38%测试，12个PR，峰值：晚10点 | 连续47天

        ## 你的这一周
        32次提交，+2.4k LOC，41%测试。峰值：晚9-11点。
        最大发货：Cookie 导入系统（浏览器解密 + 选择器 UI）。
        你做得好的：用一次专注的推送完成了含加密、UI和18个单元测试的完整功能……

        ## 团队分解

        ### Alice
        12次提交，专注在 app/services/。每个 PR 都在 200 LOC 以下——自律。
        成长机会：测试比 12%，在支付系统变复杂之前值得投入。

        ### Bob
        3次提交——修复了仪表板 N+1 查询。小但高影响。
        成长机会：本周只有1个活跃天——确认是否有什么阻塞。
```

结果保存到 `.context/retros/` 供下次比对趋势。

## 五、安全防护类技能

### `/careful` — 危险命令安全拦截

**适用**: 在生产环境附近操作，或需要安全网时  
**说一句**: "be careful" 或 `/careful` 即可激活

拦截以下危险操作（可在每次警告时选择覆盖）：

| 命令 | 风险 |
|------|------|
| `rm -rf` / `rm -r` | 递归删除文件 |
| `DROP TABLE` / `DROP DATABASE` / `TRUNCATE` | 数据库数据丢失 |
| `git push --force` / `git push -f` | 重写历史 |
| `git reset --hard` | 丢弃提交 |
| `git checkout .` / `git restore .` | 丢弃未提交改动 |
| `kubectl delete` | 生产资源删除 |
| `docker rm -f` / `docker system prune` | 容器/镜像丢失 |

**白名单**（不触发警告）: `rm -rf node_modules`, `dist`, `.next`, `__pycache__`, `build`, `coverage` 等常规构建产物清理。

---

### `/freeze` — 限制编辑范围

**适用**: 调试时防止误改无关代码

```
/freeze src/billing
→ 编辑范围已限制在 src/billing/

[Claude 尝试修改 src/auth/middleware.ts]
→ BLOCKED — 编辑操作超出 freeze 边界（src/billing/）。跳过此修改。
```

**注意**: 只拦截 Edit 和 Write 工具。Bash 命令（如 `sed`）仍可修改边界外的文件——这是意外防护，不是安全沙箱。

`/investigate` 调试技能会自动激活 `/freeze`，将编辑范围限制在正在调查的模块。

---

### `/guard` — 最高安全模式

`/careful`（危险命令拦截）+ `/freeze`（编辑范围限制）的组合。适合在生产环境或关键系统上操作时使用。

---

### `/unfreeze` — 解除编辑限制

移除 `/freeze` 设置的边界，重新允许编辑所有目录。如需重新设置边界，再次运行 `/freeze`。

---

### `/gstack-upgrade` — 升级 gstack

检测安装方式（全局 `~/.claude/skills/gstack` 或项目内 `.claude/skills/gstack`），执行升级，若存在双重安装则同步两处，显示新版本变更。

```
/gstack-upgrade

Claude: 当前版本：0.7.4
        最新版本：0.8.2

        新增内容：
        - Browse handoff（验证码/认证墙处理）
        - /codex 多 AI 第二意见
        - /qa 现在始终使用浏览器
        - 安全技能：/careful, /freeze, /guard

        已升级至 0.8.2。全局和项目安装均已同步。
```

设置 `auto_upgrade: true` 可在每次会话开始时静默升级。

## 六、附加集成：Greptile 自动代码审查

[Greptile](https://greptile.com) 是一个自动审查 PR 的服务，能捕获通过 CI 但会在生产爆炸的真实 Bug。

gstack 的 `/review` 和 `/ship` 集成了 Greptile：自动读取其评论，分类处理：
- **有效问题** → 加入关键发现，修复后发布
- **已修复问题** → 自动回复确认
- **误报** → 询问你确认后，回复解释为何是误报

每个确认的误报都保存到 `~/.gstack/greptile-history.md`，后续运行自动跳过已知误报模式。`/retro` 还会跟踪 Greptile 的信噪比随时间的变化。

**开启方式**: 在 GitHub 仓库安装 [greptile.com](https://greptile.com)，约 30 秒，gstack 自动接收其评论，无需额外配置。
