# AI 自动化子服务实施手册 (基于标准模板)

> **版本**: 1.0  
> **日期**: 2026-03-24  
> **状态**: 标准模板推荐流程  

---

## 0. 文档定位与目标

本文档是基于 `auto-project-template` 开发全新 AI 自动化子服务（Data Plane）的 **标准实施手册**。
它的目标是指导全栈开发者或 AI 助手（Agents），以最快速度和最高标准，将一段粗略的新业务构想转化为可运行、且能 **完全无缝注册对接至 AI Auto Hub (主控枢纽)** 的正式业务节点。

- 本文档**不是**通用开发教程，而是强约束的**执行工作流**。
- 它规定了在开发一个 Hub 挂载子服务时，必须经过哪些评审阶段（通过 `gstack` 技能实现）以及应当输出哪些中间文档产物。

---

## 1. 实施原则

1. **绝对复用，聚焦业务**：鉴权拦截、Token 校验、视觉外框（导航、侧边栏）、基础颜色令牌（Tokens）均通过模板/Hub 体系复用。禁止在子项目中重复开发独立的登录页或账号系统。
2. **文档驱动架构**：代码落地前，必须先生成服务专属的 PRD、工程架构设计与状态字段说明。
3. **Hub 一致性先行**：在写复杂的 AI 流水线前，优先确保该服务的注册机制、心跳上报机制和 `/healthz` 探活接口可以与 Hub 的标准协议对接。
4. **Agent 间文件通信**：所有的审查、生成结果必须写回 `docs/` 指定目录，严禁在聊天界面输出长篇长论。后续 Agent 将从文件读取上下文。

---

## 2. 交付范围

一次标准的子服务化开发落地，最终交付物必须包含：

### 2.1 规范文档层
- 子服务专属 `PRD` (`docs/PRD/`)
- 子服务工程方案（包含 API 设计与状态机概念）(`docs/architecture/`)
- 子服务设计验证 (`docs/design/`)

### 2.2 核心代码层
- **业务视图前端**：集成好所需 Tailwind 样式的 Vue3 页面。
- **业务后端 API**：FastAPI 承载的模型调用、Prompt 模板解析逻辑与数据持久化。
- **Hub 对接件**：包含读取 `HUB_API_URL` 环境变量的拦截器、初始化时的自动注册（Registration）脚本及高可用探活接口。

---

## 3. 实施门禁

新子服务并入主网（或上线投产）前，必须依次通过以下门禁把控：
1. **边界审查**：澄清业务与底座 Hub 的职责划分。
2. **架构审查**：业务表结构、外部 AI 服务接口调用速率与限制是否合理。
3. **视觉一致性**：业务内部特有的交互是否严重偏离现有规范。
4. **测试与 QA**：未登录重定向是否生效，心跳机制是否工作。

## 4. 详细实施工作流 (Detailed Implementation Workflow)

以下环节展示了如何一步步利用 `gstack` 核心技能栈打通一个子服务。
**使用者在开发该阶段时，请直接复制对应区块中的 `推荐提示词` 执行。**

### A-0. 业务需求与边界确认
- **当前状态**: 未开始
- **依赖**: 无
- **输入文件**: 一句话需求或 `docs/ideas/draft.md`
- **输出文件**: `docs/PRD/service_prd.md`

**推荐提示词：**
```text
/office-hours：

请分析用户对于新 AI 自动化业务（如视频生成、自动化抓取工具等）的初步要求，需求线索可能来源于用户的输入对话或以下文件：
- docs/ideas/draft.md

请结合项目根目录下的 `README.md`，理解本模板的服务定位（不需要开发登录系统，所有登录依赖 Hub）。基于此，整理并输出该子服务的目标受众、核心功能列表和明确的不做范围内的事项。
你必须将最终的分析结果直接写入到 `docs/PRD/service_prd.md`。不要在聊天对话框中输出最终分析结果。
```

### A-1. 架构与工程方案设计
- **当前状态**: 已完成
- **依赖**: A-0 已经稳定输出
- **输入文件**: `docs/PRD/service_prd.md`，`README.md`
- **输出文件**: `docs/architecture/service_engineering_plan.md`

**推荐提示词：**
```text
/plan-eng-review：

请基于该子服务的产品需求文档（docs/PRD/service_prd.md）和核心脚手架定位（README.md），制定并输出本服务的工程实施计划。
输出内容必须包括：
1. 本地数据库表结构设计（仅限本业务域数据，不包含全局账号表）。
2. 与外部 AI 接口调用的主要逻辑或工作流编排方案。
3. 对接 AI Auto Hub 所必需的 `/healthz` 接口设计与自动注册流预设计。
4. 本地前端组件树拆解。

请务必将最终输出写入 `docs/architecture/service_engineering_plan.md`。
```

### A-2. UI 与交互边界审查
- **当前状态**: 已完成
- **依赖**: A-1 输出结束
- **输入文件**: `docs/PRD/service_prd.md`，`docs/architecture/service_engineering_plan.md`
- **输出文件**: `docs/design/service_design_guidelines.md`

**推荐提示词：**
```text
/plan-design-review：

请审查该业务线的 PRD 与工程设计。在保证外层框架完全从 Hub 继承的前提下，专注于该服务内部独特页面的交互逻辑与容错处理。
重点考虑以下场景并在文档中详细标明：
1. 如果 Hub 返回“未登录拦截”，前端表现应当是什么样？
2. 如果底层 AI 接口调用超时，业务区应该展示怎样的加载或错误降级状态？
3. 该业务页面表单校验失败的默认提示是什么？

请将审查结果及页面逻辑细则写入 `docs/design/service_design_guidelines.md`。
```

### A-3. 视觉规范落地
- **当前状态**: 已完成
- **依赖**: A-2 已经稳定输出
- **输入文件**: 一切现有文档，含 `README.md` 中指出的 Hub Tailwind-like 风格。
- **输出文件**: `DESIGN.md` 或本模块相关 CSS

**推荐提示词：**
```text
/design-consultation：

依据 `docs/design/service_design_guidelines.md` 与 AI Auto Hub `Tailwind-like` 规范，制定具体的间距、主辅色和交互过渡动画的视觉方案，写出这部分代码层面的设计文档。
严禁使用自行拍脑袋发明的杂乱颜色或风格。
```

### A-4. 业务代码具体实现
- **当前状态**: 未开始
- **依赖**: 所有前置架构设计和 PRD 均已收敛到 docs 目录
- **输入文件**: 所有的 `docs/*` 中的文档
- **输出文件**: FastAPI 核心文件，Vue3 组件，数据库模型（如 `apps/` 目录下）

**推荐提示词：**
```text
目前进展到业务代码实现：
请你使用 `/freeze` 并指定该业务逻辑所属的目录。严格保证不会误触共用的核心 Auth 组件。
然后，根据 `docs/architecture/service_engineering_plan.md` 中的指引：
1. 首先利用 python 开发对应的 BaseModel 与后台逻辑。
2. 其次完成 Vue3 的页面级代码。
遇到问题请主动调用 `/investigate`。所有依赖和报错修复后才算完成。
```

### A-5. 自动化 QA 与联调验证
- **当前状态**: 未开始
- **依赖**: A-4 完成且本服务能本地启动
- **输入文件**: 运行中的本地服务（例如 `http://localhost:8000` 或 `http://localhost:3000`）
- **输出文件**: QA 报告（由 `gstack qa` 自行写入）

**推荐提示词：**
```text
/qa http://localhost:8000

请用 Full 模式系统性测试当前子服务的前后台联动及其核心业务。
重点覆盖项：
1. 数据写入与核心 AI 接口流程闭环。
2. 空状态或加载中的渲染正确性。
3. 访客无凭证访问受保护路由时配置的重定向行为是否生效。
4. 服务的 `/healthz` 是否 200 以及符合规范。
```

### A-6. 代码审查
- **当前状态**: 未开始
- **依赖**: A-5 完成基本正向联机。
- **输入文件**: `apps/` 下编写过的代码模块
- **输出文件**: `docs/reviews/service_review.md`

**推荐提示词：**
```text
/review

请在归档之前独立审查刚刚实现的服务模块，重点：
1. 异常处理与重试机制（特别是调用外部 AI 时）。
2. 是否所有页面调用均正确携带并传递了 Hub 状态。
3. 日志完整度。
发现高优问题请修复或通知反馈，结论写入 `docs/reviews/service_review.md`。
```

### A-7. 封版与发布归档
- **当前状态**: 未开始
- **依赖**: A-6 无阻塞型问题
- **输入文件**: 本次的验收结论及全部开发内容
- **输出文件**: 发布版本内容。

**推荐提示词：**
```text
/document-release

请根据当前子服务实现的最新代码及其 QA 结果，更新项目的主 README，提供一份将其部署以连接 AI Auto Hub 生产网的具体指引。
```
