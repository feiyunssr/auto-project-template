# 子服务页面设计审查与交互基线

- 文档状态：Reviewed
- 更新日期：2026-03-30
- 对齐基准：`/home/zero/external-cortex/Zero/ai-auto`
- 配套文档：根目录 `DESIGN.md`

## 1. 审查结论
`auto-project-template` 之前的默认呈现更像独立业务页，而不是 `ai-auto` 家族的一员。现在的设计基线已经改成：
- 模板独立运行时，自带与 `ai-auto` 一致的 Hub-style shell。
- 接入真实 Hub 后，业务区继续沿用同一色板、面板层级和状态反馈。
- 不再允许独立维护浅色后台版和 Hub 深色控制台版两套视觉结果。

## 2. 总体原则
1. 以 `ai-auto` 为唯一视觉基准。
页面的颜色、卡片、边框、抽屉、按钮和状态语义都向 `ai-auto` 对齐，不再自行定义另一套轻后台语言。

2. 子服务保留业务上下文，但不脱离 Hub 家族气质。
即便模板独立启动，也要通过侧栏、顶栏、hero 区和玻璃化面板，明确表达它属于 AI Auto Hub 体系。

3. 异常反馈留在业务区处理。
会话失效、AI 超时、校验失败和结果待复核，都只能在业务区横幅、卡片或抽屉里处理，不能切成另一套页面主题。

4. 视觉与交互一起对齐。
不仅要改颜色，也要改结构层级、按钮权重、抽屉行为和状态 badge 语义。

## 3. 页面结构要求
### 3.1 默认外壳
- 独立运行时，页面必须包含左侧导航、顶部工作区标题和底部会话信息区。
- 侧栏宽度、模糊背景、边框和 hover 层级向 `ai-auto` 看齐。
- 顶栏右上角保留上下文说明，不能退化成只有业务卡片的裸页。
- 页面组织应沿用 `router + pages + api + styles` 结构，而不是把所有内容塞进单个首页组件。

### 3.2 业务主区
- 首屏先给 hero 区，明确当前服务与 Hub 的关系。
- `ServiceStatusBanner` 放在 hero 区后面，承担全局服务状态、会话阻断和超时重试提示。
- `TaskCreateCard` 与 `TaskListCard` 为首屏并列主面板。
- `TaskDetailDrawer` 与 `AiProfileDrawer` 使用统一的右侧抽屉语义。

## 4. 状态与交互规则
### 4.1 会话失效
- 使用危险横幅，不得渲染本地登录页。
- 停止轮询与提交，保留已填表单值。
- 自动跳转 Hub 登录前，业务区需要清楚说明“当前会话已失效”。

### 4.2 AI 超时
- 区分“调用中”“自动重试中”“最终失败”三层状态。
- 自动重试中使用 `warning`，而不是直接切失败色。
- 最终失败后，详情抽屉保留最近一次成功结果或最近一次 attempt 证据。

### 4.3 表单校验
- 错误提示出现在字段旁和表单顶部，不靠全局 toast。
- 首个错误字段自动聚焦。
- 修正后应即时消除错误提示。

### 4.4 待复核
- `review_required` 不是失败，必须使用 `warning` 或中性语义。
- 不得使用红色失败文案或失败动作命名。

## 5. 文案基线
默认文案继续沿用既有业务约定，但视觉层级要遵守 `ai-auto` 风格：
- 会话失效：`当前会话已失效，已暂停提交与刷新操作`
- AI 重试中：`AI 响应较慢，系统正在重试`
- AI 最终失败：`AI 处理超时，请稍后重试或调整输入后重新提交`
- 待复核：`任务已完成，但结果仍需人工复核`

## 6. 验收清单
- 从页面第一屏就能判断该项目与 `ai-auto` 是同一设计体系。
- 不再出现独立浅色业务页。
- 侧栏、顶栏、卡片、横幅、抽屉和状态 badge 的视觉语义统一。
- 会话失效、超时重试、最终失败、待复核四类关键状态都有明确且一致的表现。
