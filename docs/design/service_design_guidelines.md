# 子服务内部页面设计审查与交互细则

- 文档状态：Reviewed
- 更新日期：2026-03-24
- 审查范围：`docs/PRD/service_prd.md`、`docs/architecture/service_engineering_plan.md`
- 设计前提：外层框架、导航、登录体系、全局布局和 Hub Shell 完全继承 Hub；本文件只定义子服务内部独特页面的业务区交互、状态反馈与容错处理。
- 配套文档：代码层视觉 token、间距、色板与过渡动画统一收敛在根目录 `DESIGN.md`，实现阶段需与本文件同时遵守。

## 1. 审查结论

本轮审查已经把原先的低分项转成了明确的设计基线，不再只是问题清单。结论是：在保持 Hub 外层框架完全继承的前提下，子服务内部页面的关键交互、异常反馈和默认文案已经收敛到可直接实施的程度。

| 维度 | 优化前 | 优化后目标分 | 本轮收敛结果 |
| --- | --- | --- | --- |
| Hub 继承边界 | 8/10 | 9/10 | 已明确未登录拦截只处理业务区阻断、状态清理和 Hub 跳转，不自建登录页。 |
| 任务主流程可理解性 | 7/10 | 9/10 | 已补齐提交中、重试中、失败、复核中等用户可感知状态。 |
| 容错与降级 | 5/10 | 9/10 | 已明确 AI 超时的三层降级、历史结果保留和局部失败原则。 |
| 表单反馈一致性 | 5/10 | 9/10 | 已统一默认校验文案、触发时机、汇总提示和首错聚焦。 |
| 回查与人工复核 | 7/10 | 9/10 | 已明确 `failed` 与 `review_required` 的视觉和语义差异。 |

当前未给出 10/10 的唯一原因，是这些规则还没有经过真实页面实现和验收用例验证，而不是文档本身仍存在空白。实现阶段必须把以下三项视为硬约束：
- 未登录拦截由 Hub 负责最终跳转，子服务只负责停止业务操作、清空敏感局部态并给出过渡提示。
- AI 超时必须区分“等待中”“自动重试中”“最终失败”三层状态，禁止退化为单条通用 toast。
- 表单校验必须同时提供字段级提示、页内汇总提示和首个错误字段聚焦。

## 2. 总体设计原则

1. 子服务不接管 Hub Shell。
业务页不得自绘登录页、顶栏、侧栏或全屏错误页；只能在 Hub 提供的内容区内渲染自己的状态。

2. 业务区优先保留上下文。
无论校验失败、接口超时还是结果失败，都应尽量保留用户已填写输入、筛选条件和已加载结果，避免用户重新输入。

3. 异常反馈分层，而不是一刀切。
字段错误用字段提示；任务级错误用卡片/横幅；服务级降级用业务区顶部状态条；禁止所有问题都用 toast 代替。

4. 能自动恢复的状态，先提示后自动处理。
例如未登录重定向、AI 超时后的自动重试，都应先告知用户当前发生了什么，再执行自动动作。

## 3. 核心场景细则

### 3.1 Hub 返回“未登录拦截”时，前端表现

目标：不自建登录页，不破坏 Hub 外层框架，同时避免用户继续操作已失效会话。

统一规则：
- 子服务收到 Hub 未登录标记、401/403 认证失效响应，或 Hub 会话查询明确返回未登录时，立即停止当前业务请求。
- 立刻清空仅存在前端内存中的敏感态：待提交 token、临时结果草稿、轮询定时器、进行中的重复提交锁。
- 不清空用户本次已填写的表单值和筛选项；这些值应保留到本次页面卸载前，便于用户重新登录后继续处理。
- 不弹出本地登录框，不渲染独立登录页；最终跳转只交给 Hub。

业务区展示规范：
- 初次进入页面即被拦截：业务区显示阻断态空白页，文案为“登录状态已失效，正在返回 Hub 登录页”。
- 用户操作过程中被拦截：业务区顶部显示红色状态横幅，正文内容区切为只读遮罩态，文案为“当前会话已失效，已暂停提交与刷新操作”。
- 横幅内提供一个主按钮：`返回登录`；同时在 1 秒内自动执行 Hub 登录跳转。
- 若 3 秒内自动跳转未发生，则保留按钮可点击，不重复弹 toast。

禁止行为：
- 禁止继续轮询任务状态。
- 禁止把未登录错误渲染成“任务失败”。
- 禁止把整个浏览器窗口替换为本地 401 页面。

### 3.2 底层 AI 接口调用超时时，业务区的加载与错误降级

目标：让用户区分“系统还在努力”“系统正在重试”“本次确实失败了”。

状态分层：
- `提交后排队/调用中`：显示骨架屏或结果占位卡，主文案为“任务已提交，正在调用 AI 处理”。
- `单次调用超时但仍会自动重试`：在业务区顶部显示黄色提示条，文案为“AI 响应较慢，系统正在重试（第 N/M 次）”；结果区继续保留加载态，不切失败页。
- `重试耗尽后最终失败`：结果区切换为失败卡片，不影响任务列表和其他区域继续使用。

展示细则：
- `TaskCreateCard` 提交成功后，提交按钮进入 loading，文案切为 `已提交，处理中...`，直至任务成功进入列表或收到失败确认。
- `TaskListCard` 中超时重试中的任务显示状态文案 `执行中`，辅文案显示 `响应较慢，正在重试`。
- `TaskDetailDrawer` 中时间线必须展示每次 attempt，至少区分 `调用中`、`超时`、`重试中`、`最终失败`。
- `ResultPreviewPanel` 在等待阶段显示结果骨架，不显示空状态文案，避免误导成“暂无结果”。
- 超时最终失败时，`FailureReasonPanel` 默认文案为“AI 处理超时，请稍后重试或调整输入后重新提交”。

降级动作：
- 优先保留已填写表单、任务号、最近一次 attempt 信息和可重试入口。
- 提供两个动作：`重试本次任务`、`返回修改参数`。
- 若已有历史成功结果，则详情抽屉继续展示最近一次成功结果，并在顶部加“当前重跑失败，以下为上一次成功结果”的提示。
- 超时失败只降级业务区，不得把整个页面切成全局异常页。

不建议的实现：
- 只用一个 toast 提示“请求超时”。
- 在第一次 timeout 时直接把任务标记为失败。
- 用空白页替代加载态，导致用户无法判断系统是否仍在运行。

### 3.3 表单校验失败时的默认提示

目标：形成统一、可批量复用的文案和触发规则，减少各字段各写一套提示。

触发规则：
- 首次进入页面不主动报错。
- 用户 blur 字段后，如不合法，立即显示字段级提示。
- 用户点击提交时，统一触发整表校验，并滚动/聚焦到首个错误字段。

默认提示文案：
- 必填输入为空：`请输入{字段名}`
- 必填选择为空：`请选择{字段名}`
- 必填上传为空：`请上传{字段名}`
- 长度超限：`{字段名}不能超过{N}个字符`
- 数量不足或超范围：`{字段名}需在{min}-{max}之间`
- 格式错误：`请输入有效的{字段名}`

页内汇总提示：
- 提交失败后，在 `TaskCreateCard` 顶部显示红色表单汇总条，默认文案为：`请先修正表单中的错误项后再提交`
- 若可以计算错误数量，推荐文案：`请先修正 {N} 项输入错误后再提交`

交互细则：
- 字段级提示优先于 toast；校验失败默认不弹全局 toast。
- 首个错误字段需要自动获得焦点；若字段在折叠区域内，应先自动展开该区域。
- 已出现错误的字段，在用户修正为合法值后应立即消失错误提示，不要求再次提交。

### 3.4 前后端交互契约补充

目标：避免实现时前后端各自理解一套状态，导致页面虽然“有处理”，但表现不一致。

推荐错误码与前端行为映射：
- `AUTH_REQUIRED`：表示 Hub 会话失效或未登录；前端必须执行 3.1 规则，而不是渲染成本地任务失败。
- `PROVIDER_TIMEOUT_RETRYING`：表示单次 AI 调用超时但仍在自动重试；前端保持加载骨架，并展示“系统正在重试”提示。
- `PROVIDER_TIMEOUT_FINAL`：表示超时且已耗尽重试；前端进入失败卡片，并提供 `重试本次任务`、`返回修改参数` 两个动作。
- `VALIDATION_ERROR`：表示请求未通过表单或业务校验；前端必须消费 `field_errors` 并生成字段提示与页内汇总提示。
- `REVIEW_REQUIRED`：表示技术执行完成但结果仍需人工判断；前端显示中性提示，不使用失败色和失败语义。

推荐返回字段：
- 任务详情接口至少返回：`status`、`error_code`、`error_message`、`retryable`、`current_attempt_no`、`max_retries`、`last_success_result`
- 表单校验失败至少返回：`field_errors`、`form_error_message`
- 未登录拦截响应至少可被前端识别为：HTTP `401/403` 或业务码 `AUTH_REQUIRED`

## 4. 页面级交互逻辑

### 4.1 `ServiceDashboardView`

- 仅负责拼装业务区，不覆盖 Hub 顶层导航和面包屑。
- 页面首次加载顺序为：会话确认 -> 健康摘要 -> 任务列表默认数据 -> 最近一次 AI 配置。
- 若会话无效，优先执行 3.1 规则，不继续请求业务数据。

### 4.2 `TaskCreateCard`

- 默认展示单一业务目标表单，不允许出现与本服务无关的多场景通用工作台。
- 提交前校验失败：留在当前卡片，显示字段错误和表单汇总条。
- 提交成功：表单进入只读 loading 态，直到后端返回任务已创建。
- 提交成功后不立即清空表单；只有用户主动点击“新建任务”或确认复位时才清空。

### 4.3 `TaskListCard`

- 空状态只用于“从未提交过任务”或“筛选后无结果”，不得复用为错误态。
- 轮询失败或接口暂时不可用时，保留上一次成功列表数据，并在卡片顶部显示轻量错误提示。
- 某单条任务超时重试中时，只更新该行状态，不阻塞整个列表操作。

### 4.4 `TaskDetailDrawer`

- 打开抽屉后优先显示任务元信息和当前状态，再渐进加载 attempts、结果和素材。
- 若结果尚未生成，用骨架屏占位；若执行失败，用失败原因面板替换结果骨架。
- `review_required` 必须使用区别于 `failed` 的中性提示，默认文案为“任务已完成，但结果仍需人工复核”。

### 4.5 `AiProfileDrawer`

- AI 参数保存失败时，不关闭抽屉，不清空用户输入。
- 若 Hub 会话在保存过程中失效，按 3.1 规则处理，并在重新登录后允许用户继续编辑未提交内容。
- 高风险参数如 `timeout_ms`、`max_retries` 修改后，应在保存成功提示中说明“仅影响后续新任务”。

## 5. 默认文案基线

为避免实现阶段出现随意文案，默认采用以下基础文案：

| 场景 | 默认文案 |
| --- | --- |
| 未登录拦截 | 登录状态已失效，正在返回 Hub 登录页 |
| 会话失效横幅 | 当前会话已失效，已暂停提交与刷新操作 |
| AI 调用中 | 任务已提交，正在调用 AI 处理 |
| AI 超时重试中 | AI 响应较慢，系统正在重试 |
| AI 最终超时失败 | AI 处理超时，请稍后重试或调整输入后重新提交 |
| 表单汇总错误 | 请先修正表单中的错误项后再提交 |

## 6. 实施验收清单

- 未登录场景下，子服务没有本地登录页，也不会把未登录误判成任务失败。
- AI 超时场景下，页面至少能区分加载中、自动重试中、最终失败三种状态。
- 表单校验失败时，字段提示、汇总提示和首错聚焦三项都生效。
- 页面所有异常都局限在业务区处理，不破坏 Hub 外层框架。
- 详情页能清楚区分 `failed` 与 `review_required`，避免把质量待确认误显示为系统失败。
